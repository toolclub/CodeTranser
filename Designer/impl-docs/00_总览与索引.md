## 0.4 编码规范(CI 硬约束)

1. **类型注解 100%**:所有函数参数 + 返回值必须写类型。`mypy --strict` 通过
2. **纯 domain 层**:`app/domain/*` 禁止 import 任何 `fastapi / sqlalchemy / redis / httpx`
3. **禁裸 dict 出入口**:Repository 和 Service 间传值用 **dataclass** 或 **pydantic model**;dict 只在 DB 序列化边界出现
4. **async 优先**:凡是有 IO 的函数都是 `async def`;纯计算可以是同步
5. **异常**:只抛 `DomainError` 子类(见 01 章);全局 handler 负责翻译
6. **日志**:只用 `structlog.get_logger(__name__)`;禁 print
7. **配置**:只从 `Settings` 对象读;禁 `os.getenv(...)` 分散在业务代码
8. **常量**:魔法数字放 `config.py` 或模块级 `Final[int] = ...`
9. **命名**:文件名 `snake_case.py`,类 `PascalCase`,函数/变量 `snake_case`,常量 `UPPER_SNAKE`
10. **docstring**:所有 public 类/方法有 docstring;复杂业务规则在 docstring 里引用设计文档章节号(如 `见 design-v5 §11.7`)

## 0.5 DI 组合根(`bootstrap.py`)

**关键设计**:
- 全应用只有一处做"new XxxImpl()",就是 `bootstrap.build_container()`
- 其他地方拿依赖都通过 `AppContainer` 或 FastAPI `Depends`
- Container 是只读 dataclass,启动时装配完毕不再变

```python
# app/bootstrap.py
from dataclasses import dataclass

@dataclass(frozen=True, slots=True)
class AppContainer:
    settings: "Settings"
    db_engine: "AsyncEngine"
    mongo_client: "AsyncIOMotorClient"
    redis: "Redis"
    llm_client: "LLMClient"
    sandbox_facade: "SandboxFacade"
    tool_registry: "NodeTemplateRegistry"
    event_bus: "RunEventBus"
    trace_sink: "TraceSink"
    pipeline: "CascadePipeline"
    workflow_facade: "WorkflowFacade"
    # ... repos / services 都在这里

def build_container(settings: "Settings") -> AppContainer: ...
```

每个子模块章节末尾会给出"**装配到 bootstrap 的代码片段**",最后合并。

## 0.8 时间与时区

- 存库全部 **UTC naive datetime**(MySQL `DATETIME(6)`、Mongo BSON Date)
- API 出入口用 **ISO-8601 UTC string**(`...Z` 结尾)
- 代码里只用 `datetime.now(timezone.utc).replace(tzinfo=None)`,不要 `utcnow()`(3.12 已废弃)

工具:`app/utils/clock.py`

```python
from datetime import datetime, timezone
def utcnow() -> datetime:
    return datetime.now(timezone.utc).replace(tzinfo=None)
```

## 0.12 前后端契约与 Phase1 架构(会话共识)

> 本节是对几次关键对齐的**最终定性说明**,后续所有章节以这里为准。

### 0.12.1 前后端分工

- **后端 = JSON 仓库 + 节点模板(=解释器)目录 + 三阶段验证器**
- **前端 = JSON 画布编辑器**(用 antv,后端不传 position)
- **图的载体是"一个大 JSON"**:节点模板 → 节点实例 → bundle → 森林;前端画、保存、加载、渲染,**纯 JSON 进 JSON 出**
- **节点模板既是前端看到的"节点种类",又是后端的"模拟器"**。两个投影:
  - `GET /api/node-templates` → 只含渲染字段(给前端画布)
  - `GET/POST/PUT /api/admin/node-templates` → 完整定义(给节点模板作者 / admin)
- **"大 JSON" ≈ 写好的代码**,用户提供输入 JSON 和期望输出 JSON,平台用解释器"跑一遍"并判定对不对

### 0.12.2 术语层级(全书统一)

```
元模板 (Meta Template)         单例;硬预置在后端 t_meta_node_template,admin 可改,不版本化
   ↓ 决定了
节点模板 (NodeTemplate / Tool)  即节点种类;设计人员按元模板填一份;后端有版本化快照
   ↓ 在画布上拖一次 = 实例化
节点实例 (NodeInstance)         小节点;对应代码片段;可归属 bundle,也可以游离(孤儿)
   ↓ 可选分组到
Bundle (节点集 / 大节点)        对应代码层的 class/function;只做分组,本身不执行;v1 不嵌套
   ↑ 但边不连 bundle,只连节点实例
边 (Edge)                       全局;src/dst 是任意节点实例(可跨 bundle)
   ↓ 从入度 0 沿边走到底 =
DAG (视图)                      计算出来,不存储
   ↓ 所有 bundle + 所有节点实例 + 所有边 =
森林 (CascadeForest)            一张图的完整 JSON;= 保存到 t_graph_version.snapshot
```

### 0.12.3 前端操作流程(给前端同学的思维导图)

```
设计人员:
  1. 打开节点模板编辑器(由元模板渲染表单)→ 填 description / input_schema / output_schema / edge_semantics → 保存一份私有节点模板
     POST /api/node-templates            (engine 强制 llm)
  或 admin 建全局
     POST /api/admin/node-templates      (engine 任意)

  2. 打开画布编辑器 → 左侧节点库 = GET /api/node-templates
  3. 拖节点模板到画布 → 生成 NodeInstance;可以拖进 Bundle 也可以游离
  4. 连线 → Edge(任意两个 NodeInstance,可跨 Bundle)
  5. 框选一个/一组节点 → Ctrl+C 复制成 JSON → Ctrl+V 粘贴(前端自动重发 ID)
  6. 保存整图 = POST /api/graphs/{id}/versions body.snapshot = 大 JSON
  7. 触发验证 = POST /api/runs { graph_version_id, scenarios: [{input, expected}] }

后端:
  8. Phase1:责任链 Handler 跑一遍(结构 + LLM 驱动 tool-call)→ valid / invalid
  9. Phase1 valid → Phase2 LLM 生成 C++ → Phase3 沙箱编译 + 跑真实用例
```

### 0.12.4 代码生成层级绑定(给 08 章)

| 森林里的东西 | 代码生成时变成 |
| --- | --- |
| **Bundle**(大节点) | 一个 **class / function**(namespace unit) |
| **Bundle 里的 NodeInstance**(小节点) | 被内联进该 class 的 **代码片段** |
| **游离 NodeInstance**(孤儿) | 独立的代码片段(v1 可以放入一个 `anonymous` class / 直接贴 main,由 08 章裁定) |
| **Edge** | 类之间或方法之间的调用 / 跳转 |

注意:节点模板定义里**不再**有 `code_hints.granularity` 字段——粒度由**结构层**决定。

### 0.12.5 两阶段验证的本质

| | 用什么"跑" | 验证对象 |
| --- | --- | --- |
| **Phase1 JSON 层** | **解释器**(NodeSimulator),软件级 | **设计对不对** |
| **Phase3 沙箱** | **编译器 + 真实机器**,硬件级 | **代码实现对不对** |

### 0.12.6 Phase1 架构(责任链 + LLM 驱动)

Phase1 = **LangGraph 子图,责任链式多 Handler**,每个 Handler 是独立 LangGraph 节点、可插拔、按 `handler_order` 排序,新增 Handler 不改其他代码。

**v1 首批两个 Handler**:

1. **structure_check**(纯 Python):无环、字段齐、符合 input_schema、边语义合法、bundle 归属合法
2. **scenario_run**(LLM 驱动):
   - LLM 拿森林结构(bundle 列表 + 游离节点 + 全局边)+ 每种节点模板的 description
   - 通过 **tool-call** 让每个 NodeSimulator 依次执行(执行器按 instance_id 在全森林范围查找,自动取 field_values)
   - 上下游 **纯 JSON 传递**(上游 output_json → 下游 input_json)
   - 跑完整森林得到 `actual_output_json`
   - 与用户提供的 `expected_output_json` 做 **字段级精确对比**
   - 失败时 LLM 辅助归因(不决定路由,只写 trace)

**未来可加的 Handler**(设计已支持,不改架构):覆盖率、不变量、数据流类型、场景合成、规约反思 ... 各自一个类即可。

### 0.12.7 LLM 在 Phase1 的双重角色

| 角色 | 在哪里用 |
| --- | --- |
| **单节点模拟器**(被调) | NodeTemplate.engine=llm 时,`LLMSimulator` 把 fields+input → output |
| **森林 Agent**(主导) | `ScenarioRunHandler` 里,通过 `agent_loop` + tool-call,决定调哪个节点、传什么 JSON、判定最终输出 |

两个角色复用同一套 `LLMClient` 装饰器链,只是**调用层级**不同。

### 0.12.8 节点模拟器契约(不可变约定)

- 每个 NodeSimulator:`run(fields, input_json, ctx) → SimResult.output`
- **纯 JSON 入 / 纯 JSON 出**
- 无隐藏状态;所有外部数据通过 `ctx.table_data` 注入
- 无状态 → 可被并发复用;无网络/磁盘 IO(除非通过 ctx)
- 新增节点类型按 03 章 §3.20 的 SOP,**框架完全不感知具体 Tool 名字**

### 0.12.9 DB 迁移约定

- **纯 SQL DDL** 放 `config/sql/NN.ddl`(两位数字 + 点 + ddl)
- `python -m app.cli migrate` 按字典序跑,已跑过的跳过,历史 DDL 改动抛错
- **不用 alembic autogenerate**(那一套和"写 DDL 迭代"模式冲突)
- SQLAlchemy ORM 只用于 CRUD,**不负责建表**
